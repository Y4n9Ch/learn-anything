## Context

The Learn Anything dashboard is a Vue 3 + Tailwind v4 SPA served by `serve.mjs`. The sidebar has two tabs per topic: Topics (session notes tree) and Exercises (exercise file tree). Quiz decks are generated by the CLI's `/learn:quiz` workflow and stored as JSON files under `.learn/topics/<topic>/quizzes/<concept-slug>/`. The frontend currently has no awareness of quizzes.

This design adds a Quiz tab, quiz file listing, and a card-based practice modal. All new UI follows the existing "Annotated Notebook" design language (red pen accent, Inter font, VitePress-style tokens). No new external dependencies are introduced.

## Goals / Non-Goals

**Goals:**

- Let users browse and launch quiz decks from the sidebar under any topic
- Provide a card-based quiz-taking experience with type-aware answer inputs
- Auto-grade objective questions (exact/accepted) on submission
- Display a results summary with score and per-question breakdown
- Animated card transitions for a polished feel, respecting reduced-motion preferences
- Serve quiz data through new API endpoints in `serve.mjs`

**Non-Goals:**

- AI-based grading of `ai_only` questions (reference answer only)
- Quiz creation or editing (handled by CLI)
- State.json updates from quiz results (CLI workflow handles that)
- Server-side persistence of quiz sessions or scores
- Mobile-optimized quiz UI (the existing dashboard is desktop-first)

## Decisions

### 1. Modal over page route for quiz practice

**Choice**: Use a `<Teleport to="body">` modal (like `SearchModal.vue`) instead of a dedicated Vue Router route.

**Rationale**: Quizzes are transient practice sessions, not persistent views. A modal keeps the user anchored in the topic context, avoids disrupting sidebar state, and lets users dismiss back to the quiz list without navigating away. This matches the existing `SearchModal` pattern.

### 2. Sidebar tab extension (3-tab layout)

**Choice**: Add "Quizzes" as a third tab button alongside "Topics" / "Exercises", reusing the existing `tabMode` ternary. Create `SidebarQuizTree.vue` mirroring the structure of `SidebarTopicTree.vue`.

**Rationale**: Three tabs are still compact and scannable. The quiz tree follows the same concept-grouped file-listing pattern as the exercise tree, minimizing cognitive overhead.

**Alternatives considered**: Dropdown or separate page — rejected because the user should be able to quickly toggle between notes, exercises, and quizzes while studying a topic.

### 3. Card transition animation approach

**Choice**: Vue `<Transition>` with CSS `transform` (translateX + scale + rotateY) and `opacity`, using `cubic-bezier(0.16, 1, 0.3, 1)` easing. Wrap in `@media (prefers-reduced-motion: reduce)` fallback.

**Rationale**: Pure CSS transitions are lightweight, framework-native, and don't require a gesture library. The rotateY + scale combo evokes physical card-flipping without being gaudy. Respecting reduced-motion is essential for accessibility.

**Alternatives considered**: Hammer.js / touch-swipe library — unnecessary for desktop-first, keyboard-navigable UI, adds a dependency.

### 4. State management for quiz sessions

**Choice**: Local component state in `QuizModal.vue`, lifted to a composable `useQuiz.ts` for reusability. No Vuex/Pinia.

**Rationale**: Quiz session state is ephemeral and scoped to one modal instance. A composable provides shared logic (fetch, grade, track answers) without introducing global state management. Matches the existing pattern (`useSearch`, `useToc`, etc.).

### 5. Quiz grading strategy

**Choice**: Grade immediately on submission of all questions. `exact` questions compare string/boolean directly. `accepted` questions match case-insensitively against `accepted_answers[]`. `ai_only` questions are marked as ungraded and show the reference answer.

**Rationale**: Immediate feedback is effective for learning. The `ai_only` type explicitly declares itself as ungradable by automation, so we surface the reference answer instead of guessing.

### 6. API design

**Choice**: Two endpoints:

- `GET /api/quizzes?topic=<slug>` — returns `{ groups: [{ concept_slug, concept_name, files: [{ filename, path }] }] }`
- `GET /api/quizzes/<topic>/<filename>` — returns full `QuizDeck` JSON, with filename including the concept-slug subdirectory path.

**Rationale**: List-then-fetch avoids sending all quiz content at once. The response structure mirrors the sidebar tree grouping.

## Risks / Trade-offs

- **No quiz JSON files exist yet** → The UI must gracefully handle empty state with a "No quizzes yet" message. The API must handle missing `quizzes/` directories.
- **Quiz JSON schema changes** → The `QuizDeck` / `QuizQuestion` types are defined in the CLI package. If the schema evolves, the frontend types must be manually kept in sync. Mitigation: the existing `validate-quiz.mjs` validator catches schema drift early.
- **Multiple quiz files per concept** → The tree groups by concept slug, listing all timestamped quiz files. Users may have many historical quiz files. Mitigation: sort newest first, limit to a reasonable count.
- **Large quiz decks** → A deck with 20+ questions may feel sluggish in the card UI. Mitigation: render one card at a time; the transition only animates the visible card.
- **Dark mode compatibility** → All new UI uses CSS custom properties from the existing design token system, inheriting both light and dark themes.

## Open Questions

- Should quiz results write back to `state.json` (confidence updates)? **Decision: No — the CLI workflow handles that separately. This UI is read-only practice.**
- Should we support keyboard shortcuts for answer selection (e.g., 1-4 for options)? **Decision: Yes — add A/B/C/D or 1-4 key bindings in the QuizCard component.**
